Astro는 **빠른**, **콘텐츠 중심** 웹사이트를 구축하기 위한 **올인원** **웹 프레임워크**입니다.

왜 다른 웹 프레임워크 대신 Astro를 선택해야 할까요? 다음은 5가지 핵심 디자인 원칙을 통해 Astro를 개발한 이유, 해결하고자 하는 문제, 그리고 Astro가 프로젝트 또는 여러분의 팀에 가장 적합한 선택인 이유를 설명합니다.

#### 아스트로는

1. [콘텐츠 중심적](#콘텐츠-중심)입니다. Astro는 콘텐츠가 풍부한 웹사이트를 위해 설계되었습니다.
2. [서버 우선주의적](#서버-우선주의)입니다. 웹사이트는 서버에서 HTML을 렌더링할 때 더 빠르게 실행됩니다.
3. [기본적으로 빠릅니다](#빠른-성능). Astro에서 느린 웹사이트를 구축하는 것은 불가능합니다.
4. [사용하기 쉽습니다](#편리함). 전문가가 아니어도 Astro로 무언가를 만들 수 있습니다.
5. [모든 기능을 갖추고 있으면서도 유연함](#완벽한-기능에-유연성까지-갖춘-astro). 100개 이상의 Astro 인터그레이션을 사용할 수 있습니다.

## 콘텐츠 중심

**아스트로는 콘텐츠가 풍부한 웹사이트를 구축하기 위해 설계되었습니다.** 여기에는 대부분의 마케팅 사이트, 출판 사이트, 문서 사이트, 블로그, 포트폴리오, 일부 전자상거래 사이트가 포함됩니다.

이와 대조적으로 대부분의 최신 웹 프레임워크는 *웹 애플리케이션*을 구축하기 위해 설계되었습니다. 이러한 프레임워크는 로그인한 관리자 대시보드, 받은 편지함, 소셜 네트워크, 할 일 목록, 심지어 [Figma](https://figma.com/) 및 [Ping](https://ping.gg/)과 같은 네이티브 애플리케이션과 같은 브라우저에서 보다 복잡한 애플리케이션과 같은 환경을 구축하는 데 가장 잘 작동합니다.

이것이 바로 Astro의 가장 중요한 차이점 중 하나입니다. 콘텐츠에 집중하는 Astro의 고유한 특성 덕분에 다른 애플리케이션 중심 웹 프레임워크에서는 구현하기 어려운 타의 추종을 불허하는 성능 기능을 제공할 수 있습니다.

:::tip
프로젝트가 두 번째 "애플리케이션" 진영에 속한다면 Astro가 프로젝트에 적합하지 않을 수 있습니다, **그래도 괜찮습니다!** [Next.js](https://nextjs.org/)에서 애플리케이션에 초점을 맞춘 Astro의 대안을 확인해 보세요.
:::


## 서버 우선주의

**Astro는 가능한 한 서버 측 렌더링을 클라이언트 측 렌더링보다 우선시합니다.** 이는 PHP, WordPress, Laravel, Ruby on Rails 등의 전통적인 서버 측 프레임워크가 수십 년간 사용해온 접근 방식과 동일합니다. 그러나 Astro를 사용하면 제 2의 서버 측 언어를 배울 필요가 없습니다. 모든 것이 여전히 HTML, CSS 및 JavaScript(또는 TypeScript) 쓸수있습니다.

이 접근 방식은 Next.js, SvelteKit, Nuxt, Remix 등의 다른 현대적인 JavaScript 웹 프레임워크와 대조적입니다. 이러한 프레임워크는 전체 웹사이트를 클라이언트 측 렌더링하고, 성능 문제를 해결하기 위해 주로 서버 측 렌더링(SSR)을 포함합니다. 이러한 접근 방식은 **싱글 페이지 앱(SPA)**이라고 불리며, Astro의 멀티 페이지 앱(MPA) 접근 방식과 대조됩니다.

SPA 모델에는 장점이 있지만, 이는 추가 복잡성과 성능 트레이드오프를 수반합니다. 이러한 트레이드오프는 페이지 성능 -- 특히 [Time to Interactive (TTI)](https://web.dev/interactive/)와 같은 중요한 지표를 포함하여 -- 에 영향을 미치며, 첫 로드 성능이 중요한 콘텐츠 중심 웹사이트에서는 별로 의미가 없습니다.

## 빠른 성능

좋은 성능은 항상 중요하지만 콘텐츠 중심 웹사이트의 경우 특히 더 중요합니다.
성능 저하로 인해 참여도, 전환율, 수익이 감소한다는 것은 이미 잘 알려진 사실입니다. 아래의 예시는 이러한 사실들을 잘 보여줍니다

- 100ms 더 빠를 때 마다 전환율 1% 증가 ([Mobify](https://web.dev/why-speed-matters/), +$380,000/년 수익)
- 50% 더 빠를 때 마다 매출 12% 증가 ([AutoAnything](https://www.digitalcommerce360.com/2010/08/19/web-accelerator-revs-conversion-and-sales-autoanything/))
- 20% 더 빠르면 전환율 10% 증가 ([Furniture Village](https://www.thinkwithgoogle.com/intl/en-gb/marketing-strategies/app-and-mobile/furniture-village-and-greenlight-slash-page-load-times-boosting-user-experience/))
- 40% 더 빠르면 가입률 15% 증가 ([Pinterest](https://medium.com/pinterest-engineering/driving-user-growth-with-performance-improvements-cfc50dafadd7))
- 850ms 더 빠르면 전환율 7% 증가 ([COOK](https://web.dev/why-speed-matters/))
- 1초씩 느려질 때 마다 사용자 10% 감소 ([BBC](https://www.creativebloq.com/features/how-the-bbc-builds-websites-that-scale))

많은 웹 프레임워크에서 개발 중에는 멋지게 보이는 웹사이트를 구축하지만 배포 후에는 매우 느리게 로드되는 경우가 많습니다. 사용자의 휴대폰이나 저사양 기기는 개발자의 노트북 속도와 같지 않기 때문에 자바스크립트가 그 원인인 경우가 많습니다.

Astro의 마법은 위에서 설명한 두 가지 가치, 즉 콘텐츠 중심과 서버 우선 MPA 아키텍처를 결합하여 절충점을 찾고 다른 프레임워크가 제공할 수 없는 기능을 제공하는 방식에 있습니다. 그 결과 모든 웹 사이트에서 즉시 사용할 수 있는 놀라운 웹 성능을 제공합니다. 우리의 목표는 **Astro를 사용하면 느린 웹사이트를 구축하는 것이 거의 불가능**하게 하는 것입니다.

Astro 웹사이트는 가장 널리 사용되는 React 웹 프레임워크로 구축된 동일한 사이트보다 [90% 적은 자바스크립트로 40% 더 빠르게 로드](https://twitter.com/t3dotgg/status/1437195415439360003)될 수 있습니다. 하지만 저희의 말만 믿지 마세요. 라이언 카르니아토(Solid.js와 Marko의 제작자)가 [말문이 막힐 정도로](https://youtu.be/2ZEMb_H-LYE?t=8163) 감탄한 Astro의 성능을 직접 확인해 보세요.


## 편리함

**아스트로의 목표는 모든 웹 개발자가 Astro에 접근할 수 있도록 하는 것입니다.** 아스트로는 기술 수준이나 과거 웹 개발 경험에 관계없이 친숙하고 쉽게 접근할 수 있도록 설계되었습니다.

이미 알고 있는 UI 컴포넌트 언어를 사용할 수 있도록 하는 것부터 시작했습니다. React, Preact, Svelte, Vue, Solid, Lit 등 여러 가지 언어를 모두 지원하여 Astro 프로젝트에서 새로운 UI 컴포넌트를 만들 수 있습니다.

또한 Astro에 훌륭한 컴포넌트 언어가 내장되어 있는지 확인하고 싶었습니다. 이를 위해 자체 `.astro` UI 언어를 만들었습니다. 이 언어는 HTML의 영향을 많이 받았습니다. 유효한 HTML 스니펫은 이미 유효한 Astro 컴포넌트입니다! 하지만 JSX 표현식(React)과 CSS 범위 지정(Svelte 및 Vue)과 같은 다른 컴포넌트 언어에서 차용한 몇 가지 인기 있는 기능도 기본적으로 결합했습니다. 또한 HTML에 가깝기 때문에 점진적 개선 및 일반적인 접근성 패턴을 오버헤드 없이 쉽게 사용할 수 있습니다.

Astro는 다른 UI 프레임워크 및 언어보다 덜 복잡하도록 설계되었습니다. 그 큰 이유 중 하나는 Astro가 브라우저가 아닌 서버에서 렌더링하도록 설계되었기 때문입니다. 즉, Hook(React), Stale Closure(역시 React), Refs(Vue), Observable(Svelte), atom, selector, reaction 또는 derivation에 대해 걱정할 필요가 없습니다. 서버에는 반응성이 없으므로 이러한 모든 복잡성이 사라집니다.

우리가 가장 좋아하는 말 중 하나가 **"복잡성에 옵트인하라"** 입니다. 특히 처음 온보딩할 때 개발자 경험에서 "필요한 복잡성"을 최대한 제거하도록 Astro를 설계했습니다. Astro에서 HTML과 CSS만으로 "Hello World" 예제 웹사이트를 구축할 수 있습니다. 그런 다음 더 강력한 것을 구축해야 할 때 새로운 기능과 API를 점진적으로 추가할 수 있습니다.

## 완벽한 기능에 유연성까지 갖춘 Astro

**Astro는 웹사이트 구축에 필요한 모든 것을 갖춘 올인원 웹 프레임워크입니다.** 컴포넌트 구문, 파일 기반 라우팅, 애셋 처리, 빌드 프로세스, 번들링, 최적화, 데이터 가져오기 등이 포함되어 있습니다. Astro의 핵심 기능 세트를 벗어나지 않고도 멋진 웹사이트를 구축할 수 있습니다.

만약 Astro를 더 많이 직접 제어하고 싶다면 [React](https://www.npmjs.com/package/@astrojs/react), [Svelte](https://www.npmjs.com/package/@astrojs/svelte), [Vue](https://www.npmjs.com/package/@astrojs/vue), [Tailwind CSS](https://www.npmjs.com/package/@astrojs/tailwind), [MDX](https://www.npmjs.com/package/@astrojs/mdx), [이미지 최적화](https://www.npmjs.com/package/@astrojs/image) 등과 같은 [100개 이상의 인터그레이션 기능](https://astro.build/integrations/)으로 Astro를 확장할 수도 있습니다. [자주 사용하는 CMS 연결](https://astro.build/integrations/)하거나 또는 [자주 사용하는 호스트에 배포](/ko/guides/deploy/)하는 것도 명령어 한 줄으로 가능합니다.

Astro는 UI에 구애받지 않으므로 **여러분이 원하는 UI 프레임워크**를 가져올 수 있습니다. React, Preact, Solid, Svelte, Vue, Lit는 모두 Astro에서 공식적으로 지원됩니다. 동일한 페이지에서 서로 다른 프레임워크를 조합하여 사용할 수도 있으므로 향후 마이그레이션이 더 쉬워지고 프로젝트가 단일 프레임워크에 종속되는 것을 방지할 수 있습니다.
