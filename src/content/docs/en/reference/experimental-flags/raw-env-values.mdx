The `experimental.rawEnvValues` flag disables coercion of `import.meta.env` values that are populated from `process.env`.

To disable coercion, set the flag to `true` in your Astro configuration:

```js title="astro.config.mjs" ins={4-6}
import { defineConfig } from "astro/config"

export default defineConfig({
  experimental: {
    rawEnvValues: true,
  }
})
```

## Usage

Currently, if an environment variable used through `import.meta.env` references a `process.env` variable whose value is `"true"`, `"false"`, `"1"`, or `"0"`, it will be coerced to respectively `true`, `false`, `1` and `0`. For example:

- If `import.meta.env.FOO` is called and the project root contains a `.env` file with `FOO=false`, the value will be `"false"` (string)
- If `import.meta.env.FOO` is called and the dev server is started with `FOO=false astro dev`, the value will be `false` (boolean)

This behavior is a holdover from when Astro supported dynamic prerender values in routes, which was [removed in Astro v5](/en/guides/upgrade-to/v5/#removed-support-for-dynamic-prerender-values-in-routes).


This experimental flag will keep the string value instead of turning into a boolean or a number. This aligns `import.meta.env`'s behavior in Astro with [Vite](https://vite.dev/guide/env-and-mode.html#env-variables).

In a future major version, Astro will switch to not coercing `import.meta.env` values by default, but you can opt in to the future behavior early using the `experimental.rawEnvValues` flag.

If you need coercion in Astro, we recommend you use [`astro:env`](/en/guides/environment-variables/).
