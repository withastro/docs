This feature allows access to the `astro:config` virtual module which exposes a non-exhaustive, serializable, type-safe version of the Astro configuration through two sub-modules.

For a complete overview, and to give feedback on this experimental API, see the [Serialized Manifest RFC](https://github.com/withastro/roadmap/blob/feat/serialised-config/proposals/0051-serialized-manifest.md).

To enable this virtual module, add the `experimental.serializeManifest` feature flag to your Astro config:

```js title="astro.config.mjs" ins={3-5}
import {defineConfig} from "astro/config"
export default defineConfig({
  experimental: {
    serializeManifest: true
  }
})
```

## Submodules

The virtual module exposes two submodules for accessing different subsets of your configuration values. This protects your information by only making some data available to the client, while keeping other data available only on the server.

## `astro:config/client`

It exposes information that are **safe** to expose to the client (browser), hence it can be used in any instance of the application. It exposes information such as `trailingSlash`, `build.format`, `i18n`, and more.

## `astro:config/server`

It exposes all information exposed by `astro:config/client`, plus information regarding the file system configuration, such as `srcDir`, `cacheDir`, `outDir`, and more. This information **isn't safe** to expose to the client (browsers), 
so attempting to use them wil raise an error.
